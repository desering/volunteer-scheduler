# Architecture Overview

This document helps you understand how the Volunteer Scheduler application is structured and how data flows through the system. Use it as a mental model for navigating and contributing to the codebase—not as a rigid prescription. Architecture evolves with project needs.

## High-Level Architecture

The Volunteer Scheduler is built as a **full-stack web application** combining several modern technologies:

- **Next.js** (App Router) - Full-stack React framework providing both frontend and backend
- **Payload CMS** - Headless CMS managing data models, database access, and admin interface
  - **PostgreSQL** - Primary database for persistent storage
- **Panda CSS** - Type-safe styling system with design tokens

### The Stack in Context

Think of the architecture in layers:

```
┌──────────────────────────────────────────┐
│   Browser (Client Components)            │  ← Interactive UI elements
├──────────────────────────────────────────┤
│   Next.js App Router                     │  ← Routing & server rendering
│   (Pages, Actions & Server Components)   │
├──────────────────────────────────────────┤
│   Feature Modules                        │  ← Domain-specific logic and UI
├──────────────────────────────────────────┤
│   Services                               │  ← Business logic layer
├──────────────────────────────────────────┤
│   Payload CMS API                        │  ← Data access & validation
├──────────────────────────────────────────┤
│   PostgreSQL Database                    │  ← Persistent storage
└──────────────────────────────────────────┘
```

Each layer has a clear responsibility. Requests—whether reading data (server components calling services) or writing data (server actions processing mutations)—flow **downward** through the layers toward the database. Responses flow **upward**, carrying data back to be rendered or confirmed to the user.

## File Structure Philosophy

The project organizes code around **features** with **shared utilities & components**, balancing feature isolation with reusability. This structure encourages developers to co-locate related code and minimize cross-feature dependencies. For detailed structure, see [File Structure Reference](../reference/file-structure.md).

### Core Directories

**`/src/app/`** - Next.js App Router pages and layouts\
&emsp;**`(scheduler)/`** - Main application routes (visible to users)\
&emsp;**`(payload)/`** - Admin panel routes (CMS interface)

 The split between `(scheduler)` and `(payload)` keeps user-facing code separate from admin code, which is mostly auto-generated by Payload CMS. The main app routes import and use feature components, while the admin routes are managed by Payload.

> [!NOTE]
> Route groups `()` organize files without affecting URLs, see [Next.js documentation](https://nextjs.org/docs/app/api-reference/file-conventions/route-groups) for details.

**`/src/collections/`** - Payload CMS schemas, see [Payload documentation](https://payloadcms.com/docs/configuration/collections) for details
- Each collection defines a data model (e.g., events, signups, users) and defines how it is stored in the database and managed in the admin panel
- Collections also define access control and hooks for reacting to data changes

**`/src/migrations/`** - Database migrations, see [Payload documentation](https://payloadcms.com/docs/database/migrations) for details
- Version-controlled schema changes
- Generated by developers by running `payload migrate:generate` after modifying collections

Use payload's CLI to generate and run migrations when modifying collections. Migrations ensure database schema stays in sync with collection definitions and provide a history of changes over time.

**`/src/features/`** - Feature-specific code (organized by domain)
- Each subdirectory represents a feature or domain (e.g., `events`, `auth`)
- Contains all code needed for that feature:
  - `actions/` - Server Actions for feature mutations
  - `components/` - Feature-specific React components
  - `hooks/` - Feature-specific React hooks
  - `services/` - Feature-specific data fetching and business logic
  - `utils/` - Feature-specific utilities and helpers
  - `types/` - Feature-specific TypeScript types
  - `index.ts` - Entry point, exports public API
- Encourages code locality and reduces app-wide imports

**`/src/shared/`** - Code shared across multiple features
- **`components/`** - Shared reusable UI components
  - `ui/` - Presentational 'dumb' components (buttons, forms, inputs, etc.)
  - Other shared components that span multiple features
- **`actions/`** - Cross-feature Server Actions
- **`utils/`** - General utilities used throughout the app
- **`providers/`** - React context providers
- **`constants/`** - App-wide constants

- **`email/`** - Email templates and configuration

### How Features and Shared Code Work Together

**When to put code in a feature**:
- Logic specific to a domain (e.g., event creation, signup management)
- Components used primarily within that feature
- Services that fetch or process feature-specific data
- Actions that handle feature mutations

**When to put code in shared**:
- Multiple features need the same utility
- Generic, reusable UI components (buttons, modals, forms)
- Cross-cutting concerns (auth, notifications, logging)
- App-wide constants and configuration

A feature can import from shared utilities and other features, but shared code should generally avoid importing from specific features (to prevent circular dependencies and maintain independence).

**Example dependency flow**:
```
/src/app/(scheduler)/events/page.tsx
  ↓ imports from
/src/features/events/components/EventOverview(.server.tsx)
  ↓ imports from
/src/features/events/services/getEvents.ts
/src/features/events/actions/createSignup.ts
/src/features/events/actions/deleteSignup.ts
  ↓ which may import from
/src/shared/utils/formatDate.ts
/src/shared/components/ui/Button.tsx
/src/lib/services/getUser.ts (auth)
```

## Data Flow Patterns

Understanding how data moves through the application is key to working effectively with this codebase. The feature-first based structure affects how data flows—each feature owns its logic, but shares infrastructure and utilities.

### Reading Data (Server-Side Rendering)

The most common pattern for displaying data:

```
1. Next.js Page (Server Component in /src/app)
   ↓
2. Service Function (e.g., getEvents)
   ↓
3. Payload CMS API (query builder)
   ↓
4. PostgreSQL Database
   ↓
5. Data returned up the chain
   ↓
6. Rendered to HTML on server
   ↓
7. Sent to browser
```

**Example**: When a user visits the home page:
- Page in `/src/app/(scheduler)/` imports a feature component (e.g., `EventOverviewServer`)
- Feature component is located in `/src/features/events/components/`
- It calls `getEvents()` from `/src/features/events/services/`
- Service queries Payload CMS with date filters
- Payload CMS translates this to SQL and queries PostgreSQL
- Events are returned, grouped, and rendered to HTML
- HTML is sent to the browser (fully rendered)

This pattern leverages React Server Components for optimal performance—no client-side API calls needed for initial render.

### Writing Data (Server Actions)

For user mutations (creating signups, updating profiles):

```
1. Client Component (user interaction)
   ↓
2. Form submission or button click
   ↓
3. Server Action (e.g., createSignup)
   ↓
4. Validation (Zod schema)
   ↓
5. Authorization check (getUser)
   ↓
6. Payload CMS API (create/update)
   ↓
7. PostgreSQL Database
   ↓
8. Response returned to client
   ↓
9. UI updates (optimistic or after confirmation)
```

**Example**: When a user signs up for an event:
- Button in client component (in `/src/features/signups/components/`) calls `createSignup()`
- `createSignup()` action is located in `/src/features/signups/actions/`
- Validates the request data (Zod schema)
- Checks if user is authenticated (may use shared utility)
- Verifies role capacity hasn't been exceeded (feature service logic)
- Creates signup record via Payload CMS
- Returns success/failure response
- Client component updates UI accordingly

Server Actions provide a direct, type-safe bridge between client interactions and server-side logic without building REST or GraphQL APIs.

### Admin Panel (Payload CMS)

A parallel data flow exists for administrative users:

```
1. Admin navigates to /admin
   ↓
2. Payload CMS admin UI loads
   ↓
3. Admin creates/edits content
   ↓
4. Payload saves directly to database
   ↓
5. Hooks and callbacks may trigger (emails, notifications)
```

The admin panel bypasses much of the custom application code, providing a content management interface directly to the database collections.

## Key Architectural Decisions

### Why Payload CMS?

Payload CMS serves multiple purposes:
- **Type-safe ORM**: Generate TypeScript types from database schema
- **Admin interface**: Instant CRUD UI for all collections
- **Access control**: Built-in authentication and authorization
- **Hooks system**: React to data changes (send emails, invalidate caches)
- **Validation**: Schema-level validation at the database boundary

This reduces boilerplate significantly—no need to build separate admin panels or API layers for common operations.

### Why Server Components as Default?

React Server Components offer several advantages:
- **Performance**: Data fetching happens on the server (closer to database)
- **Security**: Sensitive logic never reaches the client
- **SEO**: Content is rendered before reaching the browser
- **Simplicity**: No need for separate API endpoints for most reads

Client components are used selectively for interactivity (forms, modals, real-time updates).

### Why Server Actions over API Routes?

Server Actions provide:
- **Type safety**: End-to-end TypeScript, no serialization gaps
- **Colocation**: Actions live near the components that use them
- **Less boilerplate**: No need to define request/response schemas for HTTP
- **Progressive enhancement**: Forms work without JavaScript

API routes still exist for external integrations (webhooks, non-React clients).

### Styling with Panda CSS

Panda CSS provides:
- **Type-safe styles**: Autocomplete for design tokens
- **Zero runtime**: Styles extracted at build time
- **Consistent theming**: Centralized design tokens
- **Component recipes**: Reusable style patterns

CSS is generated from TypeScript code, ensuring styles and logic stay in sync.

## Component Patterns

### Server/Client Split

Many features use a pair of components:
- `ComponentName.server.tsx` - Fetches data, renders on server
- `ComponentName.client.tsx` - Handles interactivity, runs in browser

This pattern keeps data fetching server-side while enabling rich interactions.

**Example**: `EventOverview`
- Server component fetches initial events for current month
- Passes data as props to client component
- Client component manages date selection, filtering, real-time updates
- User interactions trigger new data fetches without full page reload

### Form Handling

Forms typically follow this pattern:
1. Client component renders form UI
2. Form submission calls a server action
3. Server action validates and processes
4. Returns result (success or validation errors)
5. Client component displays feedback

This provides progressive enhancement—forms can work without JavaScript via native form submission.

## Database Layer (Payload Collections)

Collections define the database schema and application data models. Each collection file specifies:

- **Fields**: Column definitions with types and validation
- **Access control**: Who can read/create/update/delete
- **Hooks**: Functions that run before/after operations
- **Admin UI**: How the collection appears in the admin panel

**Example from `events.ts`**:
```typescript
{
  slug: "events",
  fields: [
    { name: "title", type: "text", required: true },
    { name: "start_date", type: "date", required: true },
    // ... more fields
  ],
  access: {
    read: () => true, // Public read access
  },
}
```

Payload generates TypeScript types from these definitions, ensuring type safety throughout the codebase.

## Development Workflow

Understanding the architecture informs the development process:

1. **Define data model** in `/src/collections/` (or modify existing)
2. **Run migration** to update database schema
3. **Create service functions** in `/src/lib/services/` for data access
4. **Build server components** to render data
5. **Add client components** for interactivity
6. **Create server actions** for mutations
7. **Style with Panda CSS** using design tokens

Each step builds on the previous, leveraging the architectural layers.

## Observability

The application includes OpenTelemetry instrumentation for:
- Request tracing
- Performance metrics
- Error tracking

This helps understand how data flows in production and identify bottlenecks.

## Summary

The Volunteer Scheduler architecture emphasizes:
- **Server-first rendering** for performance and SEO
- **Type safety** throughout the stack
- **Clear separation** between data, logic, and presentation
- **Reduced boilerplate** via Payload CMS and Server Actions
- **Progressive enhancement** for reliability

This architecture is not immutable. As requirements evolve, patterns may change. The goal is to maintain clarity, type safety, and developer experience while adapting to new needs.

For specific implementation details, see the how-to guides. For detailed file organization, see the [file structure reference](../reference/file-structure.md).
